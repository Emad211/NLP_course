Part 1 — TF-IDF Theory (سلول‌های 7–23)

🎯 رویکرد آموزشی همه‌چیز اول دستی محاسبه می‌شه روی ۴ جمله فارسی، بعد با sklearn تأیید. دانشجو باید ریاضیات پشت API رو بفهمه.

سلول 8 — 1.1 مسئله اصلی MD

فرمول‌بندی ریاضی: نیاز به تابعی داریم که متن رو به بردار $\mathbb{R}^{|V|}$ ببره. هر بُعد = یک کلمه، مقدار = اهمیت اون کلمه برای این سند. مثال ۳ دکتر.

سلول 9–10 — 1.2 Term Frequency MD + PY

فرمول: $TF(t, d) = \text{count of } t \text{ in } d$

کد: ۴ سند فارسی ← tokenize (split) ← vocabulary (sorted unique) ← Counter هر سند ← DataFrame.

مثال: در Doc 0 «سنگ» ۲ بار اومده → TF=2. در Doc 2 «سنگ» نیست → TF=0.

سلول 11 — مشکل Raw TF MD

دو ضعف:

  1. Document length bias: دکتری با ۵۰۰ نظر → TF بالاتر برای همه کلمات (نه به‌خاطر ارتباط، بلکه حجم متن)
  2. عدم تمایز: «درد» در همه اسناد هست (عمومی) ولی «کلیه» فقط در چند سند (تشخیصی). raw TF هر دو رو یکسان وزن می‌ده.

← انگیزه IDF

سلول 12–13 — 1.3 IDF MD + PY

IDF(t) = log( (1 + N) / (1 + df(t)) ) + 1
N = تعداد کل اسناد  |  df(t) = تعداد اسنادی که ترم t در آن‌ها هست
+1 داخل log: جلوگیری از تقسیم بر صفر  |  +1 بیرون log: IDF هرگز صفر نشه
لگاریتم: اگه «سنگ» در ۱ از ۱۰۰۰ سند باشه، IDF=1000 بدون log خیلی بزرگه. log اون رو compress می‌کنه.

کد: برای هر ترم → df حساب کن → IDF حساب کن → DataFrame نمایش بده. بیشترین IDF = کمترین df (نادرترین ترم = informative‌ترین).

سلول 14–15 — 1.4 TF × IDF MD + PY

TF-IDF(t, d) = TF(t, d) × IDF(t)

ضرب ساده. خروجی: ماتریس وزن‌دار. ترم‌های نادر و مرتبط → وزن بالا. ترم‌های عمومی → وزن متوسط. غایب → صفر.

سلول 16–17 — 1.5 Sublinear TF MD + PY

tf_sub(t, d) = 1 + log(tf)   if tf > 0,   else 0

مشکل: آیا tf=100 واقعاً ۱۰ برابر مرتبط‌تر از tf=10 هست؟ نه. رابطه خطی نیست.

جدول compression: tf=1→1.0, tf=10→3.3, tf=100→5.6, tf=1000→7.9. یعنی ۱۰۰ برابر بیشتر فقط ۵.۶ برابر وزن بیشتر.

چرا مهمه: بعضی دکترها ۵۰۰ نظر دارن و بعضی ۵ تا. بدون sublinear TF، دکترهای پرنظر همیشه بالا میان.

کد: نمودار مقایسه raw TF vs sublinear TF.

سلول 18–19 — 1.6 L2 Norm + Cosine Similarity MD + PY

مشکل اندازه: حتی با sublinear TF، اسناد بلندتر بردارهای بزرگ‌تری دارن.

v̂ = v / ‖v‖₂ = v / √(Σᵢ vᵢ²)
بعد از L2 norm، طول هر بردار = ۱. حالا فقط جهت (محتوا) مقایسه می‌شه نه اندازه (طول).
cosine_sim(q, d) = q · d / (‖q‖ × ‖d‖) = q · d    (وقتی هر دو normalized هستند)
1.0 = کاملاً مشابه  |  0.0 = هیچ ترم مشترکی  |  منفی نمی‌شه (TF-IDF همیشه ≥ 0)
چرا cosine نه Euclidean؟ Euclidean تحت تأثیر magnitude (طول سند) هست. Cosine فقط angle (محتوا) رو می‌بینه.

کد: L2 normalize ماتریس TF-IDF ← ساخت بردار query «سنگ کلیه» ← dot product با هر سند ← ranking. Doc 0 بالاترین sim (هر دو ترم).

Part 2 — Medical Stopwords Pipeline (سلول‌های 24–34)

سلول 24 — Why Domain Stopwords? MD

مشکل: در نوت‌بوک Preprocessing، stopwords فارسی عمومی (از, به, که, ...) حذف شدن. اما در corpus نظرات پزشکی، یه لایه نویز جدید هست:

  • «عالی» در ۱۵,۰۰۰+ نظر — هیچ اطلاعات تشخیصی نداره
  • «دکتر» در تقریباً همه نظرات — قدرت تمایز صفر
  • «مطب», «نوبت» — خدماتی، نه تشخیصی

اگه حذف نشن: query «سنگ کلیه» ممکنه دکتری رو بیاره بالا که فقط نظرات «عالی» زیاد داره، نه واقعاً سنگ کلیه درمان می‌کنه.

معماری ۳ لایه: Seed (دستی ~۶۰ کلمه) → Auto-discovery (آمار-محور) → Safety Guards (محافظ)

Part 5 — Query Preprocessing (سلول‌های 42–44)

سلول 42 — Golden Rule of IR MD

⚠️ قاعده طلایی اگه corpus lowercase شده، query هم باید lowercase بشه. اگه corpus ZWNJ حذف شده، query هم. هر mismatch → ترم‌ها match نمی‌شن → بازیابی خاموشانه شکست می‌خوره.

پایپلاین ۹ مرحله‌ای (دقیقاً همون pipeline نوت‌بوک Preprocessing):

  1. حذف control chars (ZWNJ, ZWJ, direction marks, BOM)
  2. تبدیل ارقام فارسی/عربی → انگلیسی (۰→0, ١→1)
  3. نرمال‌سازی عربی→فارسی (ي→ی, ك→ک, أ→ا, ة→ه)
  4. حذف کاراکترهای غیرمجاز (emoji, symbols)
  5. کاهش تکرار حروف (عاااالی → عاالی)
  6. hazm Normalizer (فاصله‌گذاری صحیح فارسی)
  7. Domain deglue (درحال→در حال, باسلام→با سلام)
  8. hazm WordTokenizer + حذف stopwords فارسی
  9. [فقط Medical] حذف stopwords پزشکی

حفظ Negation:

در مرحله ۸، stopwords فارسی حذف می‌شن به‌جز: {نه, نخیر, نیست, نمی, نبود, نشد, بدون}. چرا؟ «راضی نیستم» بدون «نیست» → «راضی» — معنی عوض شد!

Part 6 — Retrieval (سلول‌های 45–48)

سلول 45 — How it works MD

۳ مرحله: (1) vectorize query با همون fitted vectorizer (2) cosine sim با همه doctor vectors (3) sort + top-K.

سرعت: چون هر دو L2-normalized هستن، cosine sim = dot product = ضرب ماتریس sparse. برای هزاران دکتر: milliseconds.

سلول 46 — retrieve() + display_results() PY

def retrieve(idx, query_text, top_k=10):
    qv = idx["vectorizer"].transform([query_text])
    if qv.nnz == 0: return []    # تمام ترم‌ها OOV → هیچ نتیجه‌ای
    sims = cosine_similarity(qv, idx["X"]).ravel()
    top_i = np.argsort(-sims)[:top_k]
    return [(doctor_ids[i], sims[i]) for i in top_i]

qv.nnz == 0: اگه هیچ ترم کوئری در vocabulary نباشه (OOV) → بردار صفره → نمی‌تونیم جستجو کنیم.

Part 7 — Recommendation-Aware Reranking (سلول‌های 49–53)

سلول 49 — The Problem + Formula MD

مشکل: cosine sim فقط شباهت متنی رو اندازه می‌گیره. دکتری ممکنه نظرات زیادی درباره «سنگ کلیه» داشته باشه ولی بیمارا ازش شاکی باشن (تشخیص اشتباه, عمل ناموفق). فقط sim بالا ≠ دکتر خوب.

final_score = sim × smooth_recommend × log(1 + q_support)
sim = شباهت cosine (آیا متن دکتر match می‌کنه؟)
q_support = چند نظر این دکتر حداقل ۱ ترم کوئری دارن
q_recommend = از اون نظرات، چندتا rate ≥ 4 دارن (توصیه)
smooth_recommend = (q_recommend + 1) / (q_support + 2) — Bayesian smoothing

Bayesian Smoothing — Beta(1,1) Prior:

دکترq_rec / q_supRaw RateSmoothedکدوم بهتره؟
A1 / 1100%(1+1)/(1+2) = 66.7%—
B45 / 5090%(45+1)/(50+2) = 88.5%✅ B بهتره — شواهد بیشتر

بدون smoothing → A بالاتر (100% > 90%). ولی ما به B اطمینان بیشتری داریم.

log(1 + support):

دکتری با ۵۰ نظر مرتبط → شواهد قوی‌تر از ۲ نظر. ولی ۱۰۰ نظر نباید ۵۰× بهتر از ۲ باشه → لگاریتمی.

def comment_matches(comment_text, terms):

    return any(f" {t} " in padded for t in terms)

Token-boundary safe: بدون padding، «سنگ» ممکنه داخل «سنگین» match بشه. با " سنگ " فقط token کامل match می‌شه.

get_query_terms(): از vectorizer.build_analyzer() استفاده می‌کنه — دقیقاً همون ترم‌هایی که vectorizer می‌بینه (شامل bigrams).
